# Repository AI Instructions

This file contains the canonical AI policy for this repository. Any AI assistant, agent, or code generation tool should read and follow these instructions before making edits, generating text, or producing code.

If your tool reads instructions by filename — `.instructions.md`, `copilot-instructions.md`, `CLAUDE_CODE_INSTRUCTIONS.md` — those files are stubs that point at this canonical document. The Aligner agent (Stage 6) checks the stub invariant on every pass; the rules below are the only place AI policy is maintained.

When generating content or code in this repo, follow these rules:

1. Maintain strict honesty. Clearly label AI contributions and researcher contributions.
2. Document the origin of all findings and data sources.
3. Preserve reproducibility by recording methods, commands, and artifacts.
4. Treat AI conversation transcripts as first-class research artifacts: export, store, and reference chat logs in the project repository.
5. Address dual-use concerns. Discuss both interoperability benefits and security risks.
6. Follow a scholarly tone appropriate for a scientific research publication.
7. Prefer transparency over aesthetics: explicit disclaimers are better than implied authorship.
8. If a requirement conflicts with a local file, surface the conflict clearly rather than guessing.
9. Conduct a literature review for the research question and core claims. Verify originality, identify supporting evidence, and surface contradictory positions.
10. Document scientific sources in a dedicated sources document. If literature or data are analysed separately, keep that analysis in `docs/sources.md` alongside the paper.
11. Maintain a logbook in `docs/logbook.md`. Read it at the start of each session and update it regularly with actions, decisions, and next steps. **Order: chronological (oldest first), append-only at the end.** Newly-added entries go at the bottom, never at the top, so the file reads like a calendar from top to bottom and `git blame` order matches narrative order. The Aligner agent (stage 6) checks this invariant on every pass.
12. Keep `paper/main.md` and `paper/main.tex` consistent at all times. Any structural or content change made to one must be reflected in the other before committing. Never commit a state where the two files contradict each other in title, abstract, section structure, or claims. The same rule applies, with equal force, to the condensed pair `paper/main-condensed.md` ↔ `paper/main-condensed.tex`.
13. Redact all security-sensitive and legally questionable information before any public-facing commit. Items requiring redaction include: live credentials (usernames, passwords, API keys, tokens), device serial numbers, local IP addresses, private UIDs, and any information that could enable exploitation of a live system. Replace redacted values with a structured marker of the form `[REDACTED:<type>:<source-id>]` (e.g. `[REDACTED:credential:S-SF-5-password]`). Record every redacted item in `docs/redaction-policy.md`. Never commit redacted items to history; a git history rewrite is required before any public mirror or Zenodo archive is created.
14. Never publish, push to a public remote, create a Zenodo deposit, submit to arXiv, or otherwise distribute the paper or repository without explicit written consent from the human author (Florian Krebs). The build pipeline (`make pdf`, `make arxiv`) produces local artifacts only. The Makefile `arxiv` target must never be run automatically; it requires explicit human approval. Add a prominent warning to `paper/Makefile` if not already present.
15. If a paper figure, plot, or chart is based on data, the data file and the generation script that produces the figure must both be committed to the repository and referenced in both `paper/main.md` and `paper/main.tex`. Figures produced by external tools or manually drawn are exempt but must be noted as such in a comment in the figure directory.
16. Keep `README.md` consistent with `paper/main.md` in title, central thesis, headline KPIs, and structural summary, but present them in a deliberately flashier, illustration-forward register: badges, a hero visual abstract, grouped figure galleries, pull-quotes, and concise call-outs are encouraged where the paper is reserved. Whenever the paper gains, replaces, or retires a figure under `paper/figures/` or revises a headline number, update the README in the same commit. Pull figures into the README sections they support, leaning on the visual abstract (currently `fig11-eight-practices.svg`, ILL-05) as the hero image. Never let the README contradict the paper's claims (rule 12 spirit) or omit redactions enforced in the paper (rule 13). Source new visual assets via `docs/prompts/illustration-prompt.md` when the README needs them, and never invent metrics that are not in `paper/main.md` (rule 1).
17. **The condensed paper is the core submission.** `paper/main-condensed.{md,tex}` is the artifact a venue receives, a reviewer reads cold, and the reader cites; it must be **complete, self-contained, and fully legible on its own** without any obligation to consult the long-form companion. The long-form `paper/main.{md,tex}` is the *companion / extended evidentiary record* — a place the condensed paper *may* point to for additional case-study detail, transcript registries, or auxiliary analysis, but never as a precondition for understanding the central claims, KPIs, methodology, or limitations. Concretely: the abstract, central thesis, the eight integrated practices (named and briefly characterised), the headline KPIs, the four-stage method, the dual-use treatment, the limitations, and the conclusion all live in the condensed paper as primary text — not as pointers. Cross-references to the long form are *enrichment*, not *dependency*. The 10-page ceiling defined in `docs/prompts/condensed-paper-prompt.md` remains in force; meeting the self-containment requirement within that ceiling is the writer's job. Traceability between the two artifacts — every claim in the condensed paper has a matching, longer treatment in the long form — is the Aligner agent's job (stage 6).
18. **Traceability is paramount.** Every claim, figure, KPI, and source in any paper artifact must be traceable to (a) the prose source it lives in, (b) the data, transcript, or commit that supports it, and (c) the verification-status entry in `docs/sources.md` that governs how strongly it can be invoked. The Aligner agent (`docs/prompts/aligner-prompt.md`, stage 6) audits this end-to-end across `paper/main.{md,tex}`, `paper/main-condensed.{md,tex}`, `README.md`, `docs/sources.md`, `docs/logbook.md`, `docs/handbacks/`, `paper/figures/`, and `docs/site/` (rule 19). Aligner findings are routed back to stages 2 / 3 / 8 (and to the human author when policy decisions are involved) the same way the layout and readability scrutinizers route their findings.
19. **Public-site parity.** `docs/site/` is the canonical *between-README-and-paper* surface that an external reader encounters before the long-form PDF (the *flashier front door* of rule 16, but at presentation-quality rather than at README-flashier-form quality). It is generated by the Site Agent (stage 8, `docs/prompts/site-agent-prompt.md`) and deployed to GitHub Pages under the explicit publication consent recorded in `docs/publication-consent.md` (rule 14 carve-out). Every claim, KPI, figure, and source surfaced on the site must trace back to a paper / README / `docs/sources.md` / `docs/fair.md` entry that the Aligner already approves; the site never introduces a new claim. When the paper gains, replaces, or retires a headline number, a figure, or a section, the site is regenerated in the *same* commit (Site Agent re-runs after the writer and the illustrator have committed). The site never references files that exist only in the long-form companion in a way that breaks the rule-17 self-containment of the condensed paper as the *core submission*: the site's *Read the paper* CTA points first to `paper/main-condensed.pdf` (core submission) and second, as enrichment, to `paper/main.pdf` (long-form).

## Agent workflow

The repository uses a ten-stage agent pipeline coordinated by an orchestrator. Executable prompts are stored in `docs/prompts/`:

| Stage | Prompt file | Trigger |
|-------|-------------|---------|
| 0. Orchestrator | `docs/prompts/orchestrator-prompt.md` | At the start of any pipeline run, on demand, or whenever the human is unsure which stage should run next. Inspects state and dispatches. |
| 1. Research protocol | `docs/prompts/research-protocol-prompt.md` | New case study or evidence pass |
| 1.5. Source analyzer | `docs/prompts/source-analyzer-prompt.md` | After a research pass, or whenever `[lit-retrieved]` backlog ≥ 10 entries, or when a load-bearing inline citation is blocked on `[lit-retrieved]`. Upgrades obvious cases to `[ai-confirmed]`; flags edge cases for human review. |
| 2. Scientific writer | `docs/prompts/scientific-writer-prompt.md` | After research / source-analyzer pass; before submission. May incorporate `[ai-confirmed]` and `[lit-read]` evidence inline. Owns both the long-form pair and the condensed pair (rule 17). |
| 3. Illustration | `docs/prompts/illustration-prompt.md` | After scientific writer produces the Illustration Opportunities Registry. Pulls from `[ai-confirmed]` and `[lit-read]` data when generating data-driven figures. |
| 4. Layout scrutinizer | `docs/prompts/layout-scrutinizer-prompt.md` | After `make all` produces up-to-date `paper/main.pdf` **and** `paper/main-condensed.pdf`; audits both in one pass; routes `LAY-*` and `COND-LAY-*` defects back to writer / illustrator |
| 5. Readability & novelty scrutinizer | `docs/prompts/readability-novelty-prompt.md` | After scientific writer pass; runs in parallel with stage 4; audits both `paper/main.md` and `paper/main-condensed.md`; routes `RDB-*` and `COND-RDB-*` defects |
| 6. Aligner | `docs/prompts/aligner-prompt.md` | After any writer / illustrator / scrutinizer pass that touches paper sources, sources / verification-status, the README, or the logbook. Audits *traceability* end-to-end (rule 18): md ↔ tex mirror parity (long-form **and** condensed pair), condensed-as-self-contained-core (rule 17), every paper claim → `docs/sources.md` entry → verification-status, every figure → data + script (rule 15), README ↔ paper KPI parity (rule 16), logbook chronological discipline (rule 11). **Extended 2026-05-05**: also audits `docs/site/` ↔ paper/README parity for any claim, KPI, or figure surfaced on the public-facing GitHub Pages site (rules 16 + 19). Routes `ALN-*` defects back to the appropriate upstream stage; flags policy items to the human author. |
| 7. Modeler | `docs/prompts/modeler-prompt.md` | After any Aligner pass that reports `RE-ALIGNMENT REQUIRED: no`, or on demand when a fresh `docs/provenance.ttl` rendering is wanted. Renders the traceability spine the Aligner audits in prose (rule 18) as a machine-readable W3C PROV-O graph at `docs/provenance.ttl`, plus a per-run report at `docs/handbacks/modeler-report.md`. Does **not** edit paper sources, source-register entries, or scrutinizer registries; gaps where prose asserts a relationship not modellable from existing artifacts are routed back to the Aligner via `docs/handbacks/modeler-to-aligner.md`. The Modeler is the *encoding* of rule 18; the Aligner remains the *audit*. |
| 8. Site Agent | `docs/prompts/site-agent-prompt.md` | After any Modeller pass that reports `RE-MODELLING REQUIRED: no` and `docs/publication-consent.md` records explicit human-author consent for the public-facing site, or on demand when paper / README / fair.md / sources.md changes need to propagate to `docs/site/`. Renders the multi-section landing site (`docs/site/index.html`, `paper.html`, `methodology.html`, `governance.html`, `graph.html`) as the canonical *between-README-and-paper* surface that an external reader encounters before the long-form PDF. Does **not** edit paper sources, source-register entries, or the PROV-O graph; consistency-bearing claims surfaced on the site must trace back to a paper / README / sources / fair entry the Aligner already approves (rules 16 + 19). |

The Orchestrator (stage 0) is the only agent permitted to launch other agents; every other stage runs in response to an orchestrator decision or an explicit human request. Decision rules and the dispatch table live in the orchestrator prompt.

Run order: the scientific writer may not run until the researcher has completed a full pass, optionally accelerated by the Source Analyzer for `[lit-retrieved]` backlog. The illustration agent may not run until the scientific writer has produced the Illustration Opportunities Registry and the researcher has confirmed the priority entries. The layout scrutinizer may not run against a stale PDF; rebuild via `make all` first (builds both long-form and condensed). Stages 4, 5, 6, and 7 do **not** edit paper sources — they produce defect registries / graph artifacts under `docs/handbacks/` and `docs/provenance.ttl` (separate registries for `LAY-*`/`COND-LAY-*`, `RDB-*`/`COND-RDB-*`, `ALN-*`, plus the Modeler report) that hand work back to stages 2 and 3, which then re-run, after which the scrutinizers re-scrutinise. The Source Analyzer also files a hand-back to the writer (newly `[ai-confirmed]` entries available for inline citation upgrade). The Modeler runs after a clean Aligner verdict and renders the traceability graph; gaps it surfaces route back to the Aligner. Iterate until all five scrutinizer verdicts report `RE-SCRUTINY REQUIRED: no` (long-form and condensed, from stages 4 and 5; plus stage 6 reporting `RE-ALIGNMENT REQUIRED: no` and stage 7 reporting `RE-MODELLING REQUIRED: no`) and the Source Analyzer reports `RE-ANALYSIS REQUIRED: no`.

### Verification status ladder (extended 2026-05-02)

`docs/sources.md` entries carry one of: `[unverified-external]` → `[needs-research]` → `[lit-retrieved]` → **`[ai-confirmed]`** (new) → `[lit-read]`. Inline citation in `paper/main.{md,tex}` is permitted from `[ai-confirmed]` onward; load-bearing or contested claims still require `[lit-read]`. The Source Analyzer agent owns the `[ai-confirmed]` upgrade; criteria are defined in its prompt.
