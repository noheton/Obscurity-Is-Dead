Title: Feature: Add support for setting parameters (write commands) via EcoFlow Open API

Body:

## Summary

The current integration only supports **reading** data from EcoFlow PowerOcean/PowerPulse devices. This issue tracks the work to add **write support** — allowing users to set device parameters directly from Home Assistant (e.g. target charge level, AC charging current, operating mode).

---

## Background & Reverse Engineering Findings

EcoFlow exposes two distinct API surfaces:

### 1. Current integration: Legacy cloud API (read-only)
- Auth: `POST https://api.ecoflow.com/auth/login` → Bearer token
- Data fetch: `GET https://{region-host}/provider-service/user/device/detail?sn={sn}`
- Headers: `Authorization: Bearer {token}`, `product-type: {variant}`
- This endpoint has **no documented write/set equivalent**.

### 2. EcoFlow Open Platform API (read + write)
Documented in the official demo (`ecoflow-open-demo.zip`). This API uses **accessKey / secretKey** credentials and **HMAC-SHA256 request signing**.

Base URL: `https://api.ecoflow.com`

| Operation | Method | Endpoint |
|---|---|---|
| Get specific quota | `POST` | `/iot-open/sign/device/quota` |
| Set quota (write) | `PUT` | `/iot-open/sign/device/quota` |
| Get all quotas | `GET` | `/iot-open/sign/device/quota/all` |
| Device list | `GET` | `/iot-open/sign/device/list` |
| MQTT credentials | `GET` | `/iot-open/sign/certification` |

#### Request Signing Algorithm (from `HttpUtil.java` / `MyMapUtil.java` / `EncryptUtil.java`)

1. Flatten the request body into a sorted key-value map (dot-notation for nested objects, `[i]` for arrays).
2. Serialize as `key1=val1&key2=val2&...` (alphabetical order).
3. Append `&accessKey=…&nonce={random_6digit}&timestamp={epoch_ms}`.
4. Sign with **HMAC-SHA256** using `secretKey`.
5. Send `accessKey`, `nonce`, `timestamp`, `sign` as request headers.

For GET: signed query string is URL-encoded and appended to the URL.  
For PUT/POST: original JSON body goes in the body; signing only in headers.
