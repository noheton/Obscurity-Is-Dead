German law / reverse engineering — publishability
This is a genuinely complex area. Here is a realistic picture:

Legal basis
German law implements the EU Software Directive (2009/24/EC) through § 69e UrhG. This explicitly permits reverse engineering of software for the purpose of interoperability — reading API responses, observing network traffic, mapping fields — without the copyright holder's permission, provided:

You have lawful access to the software (you own the device / have an account)
The information obtained is used solely for interoperability
The information hasn't been made available by the manufacturer
What's already fine
Observing HTTP traffic between the EcoFlow app and the cloud API
Documenting the JSON field names and values you observe
Writing a HA integration that speaks to the same API
What adds risk
EcoFlow's Terms of Service — most cloud platforms forbid API scraping. ToS breach is a civil matter, not criminal, but they could terminate your account or send a cease-and-desist.
Disassembling the APK — the codebase comments mention "APK sources" extensively. § 69e UrhG permits decompilation for interoperability, but requires that you don't distribute the decompiled code itself, and only use the information derived from it. Citing APK field names in comments is borderline — keep it as documentation of observations, not extracted code.
Publishing API credentials or tokens — don't include real user tokens or secrets in fixtures.
Steps to make it publishable safely
Replace all APK-source comments with observation-based ones. Instead of APK source: CFG_SP_CHARGER_CHG_OPEN_FIELD_NUMBER → cfgSpChargerChgOpen, write observed field name: cfgSpChargerChgOpen. This removes the explicit APK-decompilation paper trail.
Add a clear DISCLAIMER.md stating: the integration was developed by observing network traffic from a lawfully-owned device; no proprietary code was copied; EcoFlow credentials are never shared; use at your own risk.
License the code under MIT or Apache 2.0 — these are the most HA-ecosystem-friendly licenses and make the interoperability intent explicit.
Keep fixtures anonymised — replace all real serial numbers with placeholders (already done with SN_INVERTERBOX01, SN_E_DEVICEBOX01, etc. ✓).
Do not replicate EcoFlow's trade secrets — don't publish authentication flows that bypass their security, or signing keys.
Contact EcoFlow (optional but smart) — some manufacturers (Shelly, Growatt) actively support community integrations. EcoFlow has a developer portal (developer.ecoflow.com). Reaching out to ask for an official API key / developer agreement turns the legal grey area into a green one.
With steps 1–4 in place, publishing on GitHub and submitting to HACS is low-risk under German law given § 69e UrhG's interoperability carve-out.
