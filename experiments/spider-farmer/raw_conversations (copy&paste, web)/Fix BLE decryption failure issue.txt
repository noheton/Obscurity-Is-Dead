The fix is pushed. Here's a summary of what was wrong and what was changed:

Root cause: decrypt_data() in ble_protocol.py only ever decrypted using the dynamic IV (inner-header bytes + 2 zero bytes). It never tried self._iv — the static configured IV.

Why it matters: The reference implementation (BLEPairingManager.py) reveals that the CB controller encrypts its responses to direct commands (getSysSta, getDevSta, setDevTimezone, etc.) using the static IV, while only unsolicited event messages use the dynamic IV. This means every command response was being decrypted with the wrong IV, producing garbage output and timing out.

The fix: decrypt_data() now tries the static IV first. If it finds no JSON braces in the result, it retries with the dynamic IV. The failure log now also includes both IVs and a hint about a potentially wrong key (since all 512+ decrypted bytes being garbage — not just the first 16 — indicates the AES key itself may be incorrect for this device, which is a configuration issue separate from the IV bug).
